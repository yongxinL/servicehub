Hermes Agent

A self-hosted, multi-profile AI agent platform that brings intelligent assistants to Discord, WhatsApp, and a web API — routed through a privacy-first, complexity-aware LLM proxy.

Nous Research Multi-profile Privacy-first routing Docker Compose

Hermes Agent is an open-source AI agent platform by Nous Research. In ServiceHub it runs as a single Docker container (agsvchermagt) that consolidates one or more gateway processes and a web dashboard under tini for correct signal forwarding. Each gateway connects outbound via WebSocket to its configured messaging platform — no inbound firewall ports required. All LLM calls pass through the LiteLLM proxy (agsvclitellm), which routes requests between the on-device Gemma 4B edge model and the MiniMax 2.7 cloud model based on content complexity and explicit user tags.

3.3 Configure and Verify Messaging Platforms

How Hermes connects to chat platforms

Each Hermes profile runs its own gateway process with its own bot credentials — gateways are per-profile, not per-host. That single fact drives every design choice below: there is no front-door router that demultiplexes one shared Discord bot or one shared WhatsApp number into different profiles based on sender. If you want isolated agents (separate memory, SOUL, skills, cron jobs) per person, each profile needs its own bot.

Discord — configuration options

PatternHow it worksProsCons
One bot per profile
(recommended for families)		 Each profile has its own Discord application and bot token. Family members @mention their own bot. DISCORD_ALLOWED_USERS locks each bot to a single Discord user ID. Full state isolation — each user gets their own memory, personality, skills. No risk of cross-talk. One Discord application per profile to create in the Developer Portal.
One shared bot for everyone Single bot/profile. Add all family Discord IDs to DISCORD_ALLOWED_USERS. Enable group_sessions_per_user: true in config.yaml so each Discord user gets a separate session in shared channels. Cheapest setup; per-user session histories are still kept apart inside the shared profile. Memory, SOUL, skills, and cron jobs are all shared across users — one agent, multiple chat threads.

WhatsApp — configuration options

ModePhone numberHow people message HermesIntended for
WHATSAPP_MODE=bot
(recommended)		 A dedicated number not yet registered with WhatsApp — Google Voice (free, US), prepaid SIM ($5–15), or a VoIP number. Anyone on WHATSAPP_ALLOWED_USERS messages the bot's number directly, like any contact. Real, multi-user, or family-member use.
WHATSAPP_MODE=self Your own existing WhatsApp account. You message yourself — WhatsApp's "Note to Self" thread is the agent channel. Solo testing only — cannot be shared with another person.
⚠

WhatsApp limitations that drive the design

  • One WhatsApp account can have at most ~4 linked devices — a hard cap from WhatsApp itself.
  • Pairing the same WhatsApp account on multiple Hermes profiles via Linked Devices is technically possible but practically broken: every paired profile receives every incoming message, so every agent replies. WhatsApp has no per-profile routing key. Don't do it.
  • There is no allowlist semantic that routes message X to profile A and message Y to profile B. WHATSAPP_ALLOWED_USERS only gates who can talk to this one profile's bot.
  • Self-chat mode is inherently single-user — only the account owner can open their own "Note to Self".

Practical patterns — pick one

#PatternWhatsApp costDiscord costIsolationUse when
1 Dedicated number + Discord bot per profile
✓ recommended		 One number per family member (Google Voice / prepaid / VoIP). One Discord application per family member. Full — separate memory, SOUL, skills, cron jobs. Each family member should have their own assistant.
2 Shared profile, allowlisted family One number total. Add every family member to WHATSAPP_ALLOWED_USERS. One bot total. Add every Discord user ID to DISCORD_ALLOWED_USERS; set group_sessions_per_user: true. Discord: separate sessions per user. WhatsApp: shared. ✓

This guide uses Pattern #1. The default profile gets the first dedicated WhatsApp number and first Discord bot; each additional family profile (Alice, Bob, …) gets its own dedicated WhatsApp number and its own Discord bot application. Default-profile setup follows immediately below; per-profile setup follows in §3.6.

Default profile — Discord

Create a Discord bot

  1. Go to discord.com/developers/applications → New Application.
  2. Under Bot, enable Message Content Intent and Server Members Intent. Copy the Bot Token. Failing to enable Message Content Intent is the #1 reason Discord bots don't respond.
  3. Under OAuth2 → URL Generator: scopes bot + applications.commands; minimum permissions: View Channels, Send Messages, Read Message History, Attach Files. Recommended additions: Send Messages in Threads, Add Reactions, Embed Links (permission integer 274878286912). Copy the URL and invite the bot to your server.
⚠

Without DISCORD_BOT_TOKEN and DISCORD_ALLOWED_USERS (or DISCORD_ALLOWED_ROLES), the gateway starts but denies all users by default. Both variables are required for the bot to respond.

Set the token and access control (default profile)

bashnano ${APPS_DATA}/hermesagent/.env
# Add (get your Discord User ID via Settings → Advanced → Developer Mode → right-click username):
DISCORD_BOT_TOKEN=<your-discord-bot-token>
DISCORD_ALLOWED_USERS=<your-discord-user-id>

docker compose restart agsvchermagt
docker compose logs agsvchermagt | grep -i discord

Expected: Gateway connected or Discord gateway ready. Send the bot a DM to confirm it replies.

Default profile — WhatsApp

Hermes connects outbound — no inbound port or public IP needed. Use a dedicated phone number per profile (Google Voice, prepaid SIM, or VoIP), not a personal number. Pairing a personal account exposes private messages to the server.

⚠

Without WHATSAPP_ALLOWED_USERS, the gateway starts but denies all incoming messages by default. Set it to the phone number(s) allowed to message this bot (country code, no +).

  2. On the bot's dedicated phone: WhatsApp → Settings → Linked Devices → Link a Device — scan the QR code.
  3. The wizard confirms pairing and saves the session.

3.5 Create Family User Profiles — Alice Example

Family members each get an isolated profile: a separate gateway process, data directory, Discord connection, WhatsApp session, memory store, and agent personality. Profiles share the same container but are fully isolated at filesystem and process level.

👩

Alice

Profile name: alice Discord User ID: 12345678 WhatsApp: +61 412 123 456 Gateway port: 8643
✓

Shortcut: init-profile.sh automates Steps 1–6 in one command. It seeds all config files from defaults, substitutes API keys and base URLs, and writes Discord/WhatsApp credentials directly — no manual file editing needed.

The script auto-detects whether it is running on the host or inside the agsvchermagt container, and resolves env vars + paths accordingly. Both invocations are equivalent:

Host (reads LITEM_APIKEY, LITEM_APIBASE, FCRW_APIURL, APPS_DATA from root .env; writes to ${APPS_DATA}/hermesagent/profiles/<name>/):
./shared/hermesagent/init-profile.sh alice --port 8643 --discord-token "<token>" --discord-user "12345678" --whatsapp-number "61412123456"

Container (reads LITELLM_API_KEY, LITELLM_API_BASE, FIRECRAWL_API_URL, FIRECRAWL_API_KEY from process env set by compose/agent.yml; writes to /opt/data/profiles/<name>/):
docker compose exec agsvchermagt init-profile.sh alice --port 8643 --discord-token "<token>" --discord-user "12345678" --whatsapp-number "61412123456"

Step 1 — Create the profile

Use the hermes profile create command inside the running container. This initialises the profile directory, config files, and the per-profile CLI alias in one step.

bash# Create a blank profile (bundled skills, fresh memory)
docker compose exec agsvchermagt hermes profile create alice

# Clone config + .env + SOUL.md from the default profile (keeps fresh memory)
docker compose exec agsvchermagt hermes profile create alice --clone

# Clone everything — config, personality, memories, sessions, skills, cron jobs
docker compose exec agsvchermagt hermes profile create alice --clone-all

# Clone config from a specific existing profile
docker compose exec agsvchermagt hermes profile create alice --clone --clone-from coder

After creation, Hermes registers a convenience alias at ~/.local/bin/alice inside the container, so you can use shorthand commands like alice chat, alice setup, and alice gateway start.

Step 2 — Declare the profile in the root .env

Add HERMES_PROFILES to the root .env at the ServiceHub project root — this is the Docker Compose environment file, not the hermesagent data .env. Docker Compose injects it into the container via compose/agent.yml, where start-gateways.sh reads it to know which gateway processes to launch on container start.

root .env# Single profile
HERMES_PROFILES="alice"

# Multiple profiles (space-separated)
HERMES_PROFILES="alice bob charles"

Step 3 — Run interactive setup

The setup command walks through API keys, model selection, and bot token configuration interactively. Run it once per profile immediately after creation.

bash# Interactive setup wizard for Alice's profile
docker compose exec -it agsvchermagt alice setup

# Equivalent using the -p flag
docker compose exec -it agsvchermagt hermes -p alice setup
ℹ

The setup wizard will prompt for DISCORD_BOT_TOKEN (Alice's bot token), the preferred model, and any other provider keys. Each profile requires its own Discord bot application — do not reuse the default profile's token. Create a new app at discord.com/developers/applications — see §3.6 for the full steps.

Step 4 — Set messaging credentials manually (if skipped in setup)

If the setup wizard was skipped or credentials need updating, append them directly to Alice's .env on the host.

⚠

How start-gateways.sh decides what to launch — important if you expected the default gateway to keep running alongside named profiles:

  • HERMES_PROFILES empty → one default gateway only (uses /opt/data directly, no per-profile subdirectory).
  • HERMES_PROFILES="alice bob" → one gateway per named profile; the default gateway is not started.
  • The dashboard always starts on port 12329 regardless.

To keep a "shared" agent running alongside per-user profiles, declare it as a named profile too — e.g. HERMES_PROFILES="shared alice bob" and create a shared profile with its own port. The default profile's /opt/data root is a single-user fallback and is intentionally skipped once named profiles are declared.

Gateway management — start, stop, restart, inspect

Each profile runs its gateway as a foreground child of start-gateways.sh (tini is PID 1). The container's systemd/launchd flow is not in play — hermes gateway install / uninstall / start (which manage OS services) are not relevant inside Docker. Use the foreground form (hermes gateway) and OS-level process controls, or restart the container for a clean reset of every gateway.

ℹ

Gateways read environment variables once at process start. After editing any profile's .env, the gateway for that profile must be restarted for the new tokens or allowlists to take effect.

What launches automatically

On container start, start-gateways.sh seeds defaults, substitutes API-key placeholders, and launches one gateway per name listed in HERMES_PROFILES (or a single default gateway if the list is empty). The dashboard always starts on port 12329. See §3.5 Step 7 for the full rule.

Common operations

ActionCommandEffect
Restart all gateways docker compose restart agsvchermagt tini → SIGTERM → start-gateways.sh shutdown handler → re-runs the script. Cleanest reset; every profile and the dashboard restart together.
Stop the whole stack docker compose stop agsvchermagt Graceful shutdown of every gateway and the dashboard.
Start the whole stack docker compose up -d agsvchermagt Same as boot: launches one gateway per HERMES_PROFILES entry, plus the dashboard.
Restart only Alice's gateway docker compose exec agsvchermagt pkill -f 'hermes -p alice gateway'
docker compose exec -d agsvchermagt hermes -p alice gateway		 Stops just Alice's process, starts a fresh one detached inside the container. Other profiles untouched. Useful when only Alice's tokens or .env changed.
Stop only Alice's gateway docker compose exec agsvchermagt pkill -f 'hermes -p alice gateway' Leaves other profile gateways and the dashboard running. Alice's API port goes offline until restarted.
Start Alice's gateway ad-hoc docker compose exec -d agsvchermagt hermes -p alice gateway Starts Alice's gateway as a detached process. Survives until container restart, at which point start-gateways.sh takes over again.
List profiles + gateway status docker compose exec agsvchermagt hermes profile list Shows every profile, its model, and reported gateway status.
Per-profile health check docker compose exec agsvchermagt alice doctor Runs Hermes' built-in health check for Alice's profile.
ℹ

The official hermes gateway subcommands (run, stop, restart, status, install, uninstall) are designed for systemd/launchd hosts. Inside this container they do not have a service to manage — the gateway is just a child process of start-gateways.sh. Use the OS-level commands in the table above instead.
